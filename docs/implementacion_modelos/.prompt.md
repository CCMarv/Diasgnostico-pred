# Prompt: Refactorización para alinear código con `implementacion_modelos`

Objetivo: ejecutar la refactorización mínima necesaria para aproximar la implementación
del repositorio a las guías y ejemplos en `docs/implementacion_modelos`, respetando los
principios de diseño fundacionales y sin romper contratos públicos.

Resumen rápido (pasos):

1. Inspección y lista de cambios (ya generada): identificar archivos y funciones a refactorizar.
2. Refactorización por pequeños commits: aplicar cambios no disruptivos y ejecutar `pytest`.
3. Añadir comentarios justificativos donde no sea recomendable o posible cambiar.
4. Actualizar documentación (`docs/copilot-instructions.md`, `docs/ROADMAP.md`) con las tareas y
   decisiones de refactorización.
5. Probar integración (creación/serialización de pipeline mínimo) y validar API localmente.

Reglas estrictas a aplicar (resumen):

- Mantener intactos los contratos públicos: firmas en `api/esquemas.py`, `inferencia/predictor.py` y `config.py`.
- Evitar cambios que introduzcan data leakage: el `Pipeline` debe seguir conteniendo el preprocesador.
- Donde no se pueda cambiar una implementación crítica, insertar un comentario en el código explicando
  la razón arquitectónica para retenerla.
- Commits pequeños y con tests; cada refactor debe acompañarse de al menos un test unitario que cubra
  la nueva ruta de código o que asegure la compatibilidad.

Lista priorizada de acciones (orden de ejecución recomendado):

1) `entrenamiento/comparador_modelos.py`
   - Extraer catálogo de modelos a una fábrica/configuración (archivo `entrenamiento/catalogo_modelos.py`) cuando sea viable.
   - Extraer lógica de ajuste (`GridSearchCV` vs `cross_val_score`) a funciones helper: `entrenamiento/ajuste.py`.
   - Mantener la firma de `entrenar_clasificacion` y `seleccionar_mejor`.

2) `entrenamiento/pipeline.py`
   - Reorganizar `ejecutar_pipeline` en pasos: carga → split → entrenar/comparar → evaluar → serializar → reportar.
   - Extraer `_extraer_probabilidad_clase_1` a utilitario compartido si procede.

3) `entrenamiento/preprocesador.py`
   - Asegurar que `OrdinalEncoder` use `handle_unknown` y documentar órdenes.
   - Verificar nombres de pasos del `Pipeline` (`preprocesador`, `clasificador`) y mantenerlos.

4) `inferencia/predictor.py`
   - Hacer la validación de entrada más tolerante (aceptar dict/JSON convertible).
   - No cambiar contrato de salida; documentar cambios en comentarios.

5) `api/main.py`
   - Extraer inicialización del predictor a una fábrica/lazy loader para permitir recarga del modelo sin reiniciar API.

6) Tests y documentación
   - Añadir tests que creen pipelines mínimas (mock estimators) y verifiquen serialización.
   - Actualizar `docs/implementacion_modelos/*` con fragmentos de refactor realizados.

Plantilla de commit y PR por tarea

- Mensaje de commit: `refactor(<módulo>): breve-descripción`.
- PR: Describir cambios, pruebas ejecutadas, y archivos con comentarios justificativos donde no se modificó.

Checklist antes de merge

- [ ] `pytest -q` pasa localmente.
- [ ] No se modificaron firmas públicas.
- [ ] Comentarios insertados donde no se cambió implementación crítica.
- [ ] `docs/ROADMAP.md` actualizado con subtareas de refactor (tickets S2-11, S2-12...).

Tiempo estimado (por prioridad)

- Paso 1 (comparador_modelos): 2–4 horas
- Paso 2 (pipeline): 1–2 horas
- Paso 3 (preprocesador): 1–2 horas
- Paso 4 (inferencia/api): 1–2 horas

Notas finales

Siempre que una modificación pueda alterar comportamiento en producción, preferir añadir una capa
compatibilidad o un aviso en `copilot-instructions.md` en lugar de cambiar la firma.
